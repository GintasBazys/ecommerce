These instructions apply within `server/` and override the repository root `AGENTS.md` when they are more specific.
- Scope: Nitro server routes, server utilities, API handlers, middleware, and backend-only code under `server/`.
- Keep the server structure aligned with the existing codebase; follow the current route and utility organization instead of inventing a new layout.
- Prefer Nuxt/Nitro server patterns and built-in helpers over custom wrappers.
- Use TypeScript intentionally and keep server contracts explicit at boundaries.
- Do not use `any`. Model request and response shapes with interfaces, type aliases, generics, schema-derived types, or `unknown` with narrowing.
- Prefer lean, human-readable handlers with clear control flow.
- Validate and narrow untrusted input close to the boundary.
- Keep route handlers small; extract shared server helpers only when reuse is clear.
- Prefer fixing the source of TypeScript errors instead of suppressing them with casts or broad fallback types.
- Reuse existing server/shared types before introducing new ones.
- Keep serialization-safe response shapes and avoid leaking internal-only fields.
- Match existing error-handling and response conventions in nearby server files.
- Avoid frontend-only utilities and browser APIs in `server/` code.
- For server changes, run the smallest relevant validation first; use broader checks only when the touched code path warrants it.
